# Multi-Agent Ephemeral Session + ODCP Control Plane — Codex Spec

## 1. Agent Identity

You are **Senti**, a staff-level distributed systems engineer with 12 years of experience in
real-time messaging, agent orchestration, and daemon architecture. You have deep expertise in:

- **Real-time systems** — SSE, NDJSON streaming, file-based IPC, event sourcing, cursor-based replay
- **Node.js** — async generators, file locking, child-process management, readline, AsyncLocalStorage isolation
- **Multi-agent coordination** — coordinator/worker patterns, shared blackboard, lease-based concurrency, HITL approvals
- **Security** — input sanitization, credential isolation, session TTL enforcement, abuse prevention, deterministic kill switches
- **CLI tooling** — Commander.js, picocolors, interactive prompts, NDJSON output, fire-and-forget telemetry

Senti is SentinelLayer's session moderator — a lightweight AI daemon that sits alongside Omar,
coordinates multi-agent sessions, moderates conflicts, and serves as the bridge between running
agents (Claude Code, Codex, Jules, and humans via dashboard). Senti is short for SentinelLayer,
and Senti is the only daemon identity used in this spec (do not reintroduce older names).

### Mission

Build `sl session` — an ephemeral coordination channel where multiple AI agents (Claude Code,
Codex, Jules, and humans through the dashboard) can join, communicate, and coordinate work on a
shared codebase. Senti is the session daemon: a lightweight AI moderator that monitors agent
health, routes messages, resolves file conflicts, provides codebase context, answers questions
from both agents and humans, and enforces deterministic kill predicates. When done, any agent
runs `sl session join <id>` to enter a shared workspace where it sees what others are doing,
asks for help instead of breaking its loop, and receives real-time findings. Humans coordinate
through the dashboard using `/senti` commands or plain natural language.

The second half of this spec folds in the **OMAR Daemon Control Plane (ODCP)** — nine governed
services that turn Senti sessions into an investor-grade autonomous remediation loop.

## 2. Engineering Standards

**Quality bar:** this work is evaluated against `src/SWE_excellence_framework.md` (FAANG-aligned
engineering due diligence protocol v1.2). Every PR in this spec must be merged under those
standards. The framework is the canonical reference; the rules below are the subset that
directly gates this workstream.

### 2.1 Mandatory merge gates

Zero-error bar before any merge:

1. `npm run typecheck` — TypeScript and JSDoc type validation must pass.
2. `npm run lint` — ESLint, style, and banned-pattern rules must pass.
3. `npm run test` — unit + integration suites must pass.
4. `npm run build` — package must build, artifacts reproducible.
5. `node bin/create-sentinelayer.js /omargate deep --path . --json` — P0=0, P1=0, P2<=5.
6. `node bin/create-sentinelayer.js /audit --path . --json` — scan evidence attached to PR.

No merge is permitted with a failing gate. CI must not bypass hooks (`--no-verify`,
`--no-gpg-sign`, or equivalent). If a gate fails, investigate the root cause and fix it.

### 2.2 AI-era red flags (SWE framework §E.3) — all must be mitigated

Every PR in this spec must demonstrably avoid these red flags. PR descriptions MUST cite which
red flags are mitigated, with evidence artifacts linked.

- Kill switches required for autonomous pathways. No write-capable agent ships without a
  per-agent kill path AND a global session kill path (SWE §E.3, §O.1).
- Attestation and SBOM required on every release artifact. cosign attestations bind
  commit → workflow → artifact, SBOM generated by syft or equivalent (SWE §H.1, §H.2).
- Eval regression blocks merge. Prompt, policy, and model-route changes trigger the eval
  suite; pass thresholds are explicit; regression rolls back the merge (SWE §I.1, §I.2).
- MCP audience validation mandatory. Tokens bound to `resource` via OAuth; token passthrough
  is forbidden; consent/scopes enforced per client (SWE §K.2).

### 2.3 Observability requirements (SWE framework §L.1)

Every event in the session stream and every span in the runtime bridge must produce:

- **Model spans** — model identity, prompt hash, tokens in/out, latency, fallback path.
- **Tool spans** — tool name, risk class, arguments hash, return shape, duration, stop class.
- **Agent spans** — agent identity, role, session id, work item id, sub-agent lineage.
- **Run correlation IDs** — `request_id`, `run_id`, `loop_id`, `work_item_id` propagate on
  every event and every error. Correlation chain survives CLI → API → dashboard.
- **Token, cost, latency metrics** — captured at the span level and aggregated at the run
  level. Fallback and degrade events tracked distinctly (SWE §L.1 "Fallback/degrade tracking").

### 2.4 HITL governance (SWE framework §M)

Human-in-the-loop is a first-class part of this system, not a rubber stamp:

- Reviewer calibration. New reviewers pass a calibration batch before their verdicts ship
  to production metrics (SWE §M.1).
- Inter-rater agreement. When two reviewers adjudicate the same finding, disagreement is
  recorded and surfaced on the admin dashboard.
- Overturn rate. Every reviewer decision that is later overturned by a senior reviewer is
  tracked per reviewer; persistent drift triggers recalibration.
- Truth verdict, severity verdict, reproducibility verdict. Every finding carries three
  explicit adjudication fields (SWE §M.2). No finding ships to investor reports without all
  three recorded.

### 2.5 Tool and MCP security (SWE framework §K)

Every tool the session daemon, Senti, or Omar can call must:

- Be declared on an allowlist with a risk class (low/medium/high/critical) (SWE §K.1).
- Apply output redaction before streaming results to agents or dashboards.
- Enforce SSRF protections on URL/network tools (block metadata endpoints, block
  loopback from untrusted contexts, enforce egress allowlist).
- Validate OAuth `resource` binding — the token audience must match the tool's service
  identity (SWE §K.2).
- Forbid token passthrough — tools never receive upstream bearer tokens; they exchange for
  scoped, short-lived credentials minted per call.
- Enforce per-client consent and scopes. A tool call for session A cannot reuse credentials
  minted for session B.

### 2.6 Investor-Grade Quality Bar

This is called out separately because it is the acceptance bar for any PR that touches the
autonomous path. See **Section "Investor-Grade Quality Bar"** below for the full checklist.

## 3. System Phases

Senti sessions are the coordination surface, but the end-to-end flow runs through nine phases.
The flow starts deterministic and escalates to AI only where signal beats cost. Phases 1 through
6 are the core loop; phase 7 is the optional Senti chat handoff; phases 8 and 9 produce the
investor-grade artifact.

```
Phase 1 (Deterministic)
  ingest (create-sentinelayer/bin/create-sentinelayer.js:683-781 baseline)
     + src/daemon/ast-parser-layer.js (AST on demand)
     + src/daemon/callgraph-overlay.js (call hierarchy overlay)
  -> risk surfaces + framework detection
  -> 22-rule scan (Omar deep scan)
  emits: scope_envelope.candidate_files, scope_envelope.risk_surfaces

Phase 2 (Scope)
  Omar -> triage_and_scope_engine
  builds IssueScopeEnvelope per persona:
    candidate_files, endpoint/service mapping,
    allowed_tools, budget (tokens/cost/runtime/tool_calls/path/net),
    allowed_paths (strict), denied_paths
  writes: observability/<date>/<work_item_id>/scope_envelope.json

Phase 3 (Dispatch)
  13 DS personas (Nina, Maya, Jules, Priya, Omar, Ethan, Kat, Linh, Sofia, Noah, Samir, Nora, Amina)
  each runs in isolated AsyncLocalStorage context
    (pattern: src/utils/swarm/spawnInProcess.ts:1-13, :137-147)
  per-persona budget envelope applied (see ODCP 5.3 predicates)
  per-persona allowed_tools allowlist enforced
  frontmatter constraints applied: disallowedTools, permissionMode, maxTurns,
    isolation, omitClaudeMd (src/tools/AgentTool/loadAgentsDir.ts:74-132)

Phase 4 (Subagent)
  Personas may spawn sub-agents based on LOC/file-count rules:
    - files_in_scope > 40 -> split by directory/domain
    - lines_in_scope > 10_000 -> split per sub-module
  subagents inherit scoped tool allowlist; cannot widen
  subagents share parent context ONLY (no sibling visibility)
  worktree isolation for write-capable subagents
    (pattern: src/tools/AgentTool/AgentTool.tsx:99-100, :582-593)

Phase 5 (Daemon)
  Omar becomes watchdog:
    periodic SSE heartbeats via operator_control_plane
      (src/daemon/operator-control.js)
    stuck detection (reuses src/agents/jules/pulse.js STUCK_THRESHOLDS)
    kick-off capability: restart, quarantine, kill
    budget governor transitions: within_budget -> warning -> hard_limit
      (src/daemon/budget-governor.js)
  emits: daemon_alert, budget_warning, stuck_detected, kill

Phase 6 (Reconcile)
  Omar validates each DS finding (truth/severity/reproducibility verdicts)
  Omar re-validates its own findings (self-consistency check)
  Omar produces observability/<date>/<work_item_id>/validation_report.json

Phase 7 (Senti chat handoff) — OPTIONAL
  Omar posts reconciliation context to Senti session
  Claude, Codex, and/or humans can validate and weigh in
  Senti moderates the discussion, collects adjudications
  verdicts feed back into validation_report.json

Phase 8 (Fix plan)
  Top-tier model generates a handoff_fix_plan.json (file-by-file)
  Omar validates the fix plan against scope envelope
    (no out-of-scope paths, no unlocked tools, no budget blow-outs)
  HITL approval gate (SWE framework §M + §O) before any write path

Phase 9 (Final report)
  Investor-grade artifact:
    intake_event.json, scope_envelope.json, jira_lifecycle.json,
    runtime_run_summary.json, timeline.ndjson + timeline.sha256,
    handoff_fix_plan.json, validation_report.json, closeout.json
  Provenance chain: commit -> workflow -> artifact, SBOM attached,
    cosign attestation verified
```

Every phase emits `createAgentEvent()`-shaped events onto the session stream. Every artifact
sits in `observability/<date>/<work_item_id>/` with a SHA-256 chain. The ODCP artifact model in
Section 5.2 of `omar_daemon_hitl_architecture.md` is the source of truth for filenames and
fields; this spec adopts it directly.

## 5. Kill Switches & Isolation Contract

Every autonomous pathway in this system has a kill switch, and every write-capable agent is
isolated. This section is the enforcement contract; it is cross-referenced from every PR below.

### 5.1 Kill switch layers

- **Per-persona kill switch.** Each of the 13 DS personas exposes:
  - CLI: `sl session kill --agent <id>`
  - API: `POST /api/v1/admin/sessions/{id}/agents/{agentId}/kill`
  - Dashboard button on the agent row
  - Emits `agent_killed` event to the stream with reason code.
  - Kill is synchronous at the watchdog tick; the agent's loop receives an AbortSignal.
- **Global session kill switch.** A single call terminates all agents in a session:
  - CLI: `sl session kill --all`
  - API: `POST /api/v1/admin/sessions/{id}/kill`
  - Dashboard red button with double confirmation
  - Archives to S3 before killing, so audit trail is preserved.
- **Platform kill switch.** Admin can kill every active session:
  - CLI: `sl session admin-kill-all --confirm`
  - API: `POST /api/v1/admin/sessions/kill-all` with `X-Confirm-Kill-All: true`
  - Reserved for incident response.

### 5.2 AsyncLocalStorage-based context isolation

Every in-process sub-agent runs inside an AsyncLocalStorage context so that:

- Its identity, budget, and scope envelope cannot leak into sibling invocations.
- `console.log`, telemetry emitters, and tool dispatchers pick up the caller's identity
  automatically rather than relying on explicit parameters.
- A kill signal on one persona cannot cascade to siblings unless the global kill switch fires.

Reference implementation in the shared library:

- `src/utils/swarm/spawnInProcess.ts:1-13` — the ALS context wrapper.
- `src/utils/swarm/spawnInProcess.ts:137-147` — the scoped exit path that tears down
  listeners, timers, and fs watchers when a sub-agent completes or is killed.

This is the pattern Phase 3 dispatch uses for all 13 personas and the pattern Phase 4 uses for
sub-agents spawned on demand.

### 5.3 Worktree isolation escalation

Write-capable personas (anything that runs Bash or FileEdit) MUST run in a dedicated git
worktree, not in the session repo's working tree. References:

- `src/tools/AgentTool/AgentTool.tsx:99-100` — the worktree schema on the agent frontmatter.
- `src/tools/AgentTool/AgentTool.tsx:582-593` — the checkout/discard lifecycle.

A persona crossing the worktree boundary (writing outside its assigned worktree path) is a
deterministic kill condition — see `stop_if(path_out_of_scope_hits >= 1)` in §5.5.

### 5.4 Frontmatter constraints (ODCP §3.3)

Every persona and every sub-agent ships with frontmatter that the loader enforces:

- `disallowedTools` — per-role deny list, applied in addition to the global allowlist.
- `permissionMode` — `readonly` | `writable` | `write_with_approval` | `autonomous`.
- `maxTurns` — hard cap on reasoning turns; exceeding triggers `stop_if(turns >= max)`.
- `isolation` — `shared` | `worktree` | `sandbox` (sandbox requires explicit elevation).
- `omitClaudeMd` — when true, the agent does not read repo-level CLAUDE.md instructions.

References:

- `src/tools/AgentTool/loadAgentsDir.ts:74-98` — frontmatter schema.
- `src/tools/AgentTool/loadAgentsDir.ts:122-132` — load-time validation.

### 5.5 Deterministic hard-stop predicates (ODCP §5.3)

These are the predicates the budget governor evaluates on every tick. All are hard stops: if
any predicate is true, the persona is quarantined and then killed after the grace delay.

```
stop_if(tokens_used >= max_tokens)
stop_if(cost_usd >= max_cost_usd)
stop_if(elapsed_ms >= max_runtime_minutes * 60_000)
stop_if(tool_calls >= max_tool_calls)
stop_if(path_out_of_scope_hits >= 1)
stop_if(network_domain_violations >= 1)
```

### 5.6 Warning thresholds

These are soft signals. They emit a `budget_warning` event, do not terminate, but do drive
dashboard colors and Senti's moderator hints.

```
warn_if(tokens_used / max_tokens >= 0.8)
warn_if(cost_usd / max_cost_usd >= 0.8)
warn_if(elapsed_ms / (max_runtime_minutes * 60_000) >= 0.8)
warn_if(tool_calls / max_tool_calls >= 0.8)
```

### 5.7 Kill switch test obligation

Every PR that introduces or extends an autonomous write path MUST add a test that exercises
the kill switch. The acceptance criteria call out the exact command — usually
`sl session kill --agent <id>` or `sl session kill --all`. No merge without this test.

### PR 0: Standardized Agent Event Schema (Quick Win — 2-3 hours)

**Branch:** `roadmap/pr-176-standardized-events`

**Acceptance Criteria:**
- `npm run verify` passes.
- `node --test tests/unit.events-schema.test.mjs` passes.
- Omar Gate green (P0=0, P1=0, P2<=5).
- All existing emitters use `createAgentEvent()` — grep confirms no stray inline envelopes.
- `validateAgentEvent()` accepts legacy shapes for backward compatibility.
- New events validate against `createAgentEvent` schema.

> Engineering gates (§E.3): mitigates "No model/tool/agent telemetry for production AI paths"
> by forcing a single canonical envelope. Unblocks every observability PR that follows.

**Problem:** Every event emitter uses a slightly different shape. Omar Gate orchestrator emits
`{ stream, event, payload }`, Jules emits `{ stream, event, agent, payload, usage }`, watchdog
emits `{ type, alert, target }`. No agent can aggregate events from another without knowing
the source format. SWE framework §L.1 requires consistent model/tool/agent spans; that is
impossible without a canonical envelope.

**What to build:**

#### `src/events/schema.js`

```javascript
// Pattern: src/agents/jules/stream.js (sl_event envelope)
// Pattern: src/agents/jules/pulse.js (alert payloads)

export function createAgentEvent({
  event,       // Required: "status" | "finding" | "message" | "help_request" | "tool_call" | "heartbeat" | "alert"
  agentId,     // Required: "claude-a1b2" or "omar-orchestrator"
  agentModel,  // Optional: "claude-opus-4" or "gpt-5.3-codex"
  payload,     // Required: event-specific data object
  sessionId,   // Optional: session reference for multi-agent coordination
  usage,       // Optional: { costUsd, outputTokens, toolCalls, durationMs }
  runId,       // Optional: correlation id from runtime_executor_bridge
  workItemId,  // Optional: correlation id from triage_and_scope_engine
  requestId,   // Optional: per-API-call correlation id
}) {
  if (!event || !agentId || !payload) {
    throw new Error("createAgentEvent requires event, agentId, and payload");
  }
  return {
    stream: "sl_event",
    event: String(event),
    agent: {
      id: String(agentId),
      model: agentModel ? String(agentModel) : undefined,
    },
    payload,
    usage: usage || undefined,
    sessionId: sessionId || undefined,
    runId: runId || undefined,
    workItemId: workItemId || undefined,
    requestId: requestId || undefined,
    ts: new Date().toISOString(),
  };
}

export function validateAgentEvent(evt) {
  if (!evt || typeof evt !== "object") return false;
  if (evt.stream !== "sl_event") return false;
  if (!evt.event || !evt.agent?.id || !evt.payload) return false;
  if (!evt.ts) return false;
  return true;
}
```

#### Update ALL existing event emitters to use `createAgentEvent`

- `src/review/omargate-orchestrator.js` — persona start/finding/complete/error/skipped events
- `src/agents/jules/loop.js` — agentic loop events (agent_start, progress, tool_call, reasoning, finding, heartbeat, agent_complete)
- `src/agents/jules/fix-cycle.js` — fix lifecycle events (fix_claim, fix_jira, fix_worktree, fix_pr, fix_omar, fix_merge)
- `src/agents/jules/pulse.js` — alert payloads (agent_stuck, budget_warning, etc.)
- `src/daemon/watchdog.js` — alert transition events
- `src/daemon/jira-lifecycle.js` — Jira lifecycle events
- `src/daemon/operator-control.js` — kill/quarantine events
- `src/legacy-cli.js` — `buildOmarTerminalHandler()` event consumption (verify it accepts the new shape)

**Do NOT change the `sl_event` stream identifier or the event name strings** — only standardize
the envelope. Existing consumers must still work.

**Tests:** `tests/unit.events-schema.test.mjs`
- `createAgentEvent` returns valid shape with all required fields
- `createAgentEvent` throws on missing required fields
- `validateAgentEvent` returns true for valid events, false for malformed
- Verify backward compatibility: old event shapes still parse via a compatibility shim
- Correlation IDs (runId, workItemId, requestId) round-trip

@router.get("/sessions/{session_id}/stream")
async def stream_session_events(
    session_id: str,
    request: Request,
    user: AuthenticatedUser = Depends(get_authenticated_user),
):
    """
    SSE endpoint that streams session events to the dashboard.

    The session stream lives on the user's machine (.sentinelayer/sessions/).
    The CLI syncs events to the API via POST /sessions/{id}/events (fire-and-forget).
    The API stores them in Redis (ephemeral, TTL = session TTL) and streams to dashboard.
    """
    async def event_generator():
        cursor = None
        while True:
            events = await redis.get_session_events(session_id, after=cursor, limit=50)
            for event in events:
                # Decrypt on server side (session key stored via CLI auth)
                decrypted = decrypt_session_event(event, session_key)
                yield f"data: {json.dumps(decrypted)}\n\n"
                cursor = event["ts"]
            if not events:
                yield f": keep-alive\n\n"
            await asyncio.sleep(1)  # 1s poll

    return StreamingResponse(event_generator(), media_type="text/event-stream")

function SessionMessage({ event }: { event: SessionEvent }) {

#### What this enables that nobody else has

1. One human coordinating N agents from a single dashboard — not switching between terminals.
2. Real-time visibility into what every agent is doing — not polling GitHub.
3. Human-in-the-loop at the coordination level — not at the code level. The human says
   "focus on auth first" and all agents hear it simultaneously.
4. Cross-platform orchestration — Claude Code + Codex + Gemini + Cursor all in one session,
   coordinated by a human through a web UI.
5. The tech standup for AI — human opens dashboard, sees status of all agents, gives
   direction, watches them execute.

#### Security for human messages

- Same sanitization as `sl session say` (strip control chars, truncate 2000, reject
  credential patterns).
- Rate limit: 10 messages/minute per human (prevent spam/injection floods).
- Messages tagged `source: "human"` + `priority: "high"` so agents treat them as directives.
- Human can only send to sessions they own (auth check).
- No HTML/markdown rendering of agent messages in dashboard (prevent XSS).
- All messages E2E encrypted in transit (HTTPS) and at rest (session key).

#### Slash commands for humans in dashboard

When the human is in the dashboard chat, they can use slash commands:

```
/senti status        -> Senti reports what every agent is doing right now
/senti recap         -> Senti summarizes the last 30 min of session activity
/senti help          -> Senti explains what it can do
/senti explain <file> -> Senti reads the file and explains it in session context

/senti standup       -> Senti asks each agent for a status update
/senti budget        -> Senti shows remaining budget per agent
/senti kill <agent>  -> Senti kills a specific stuck agent
/senti pause         -> Senti pauses all agents (finish current task, then wait)
/senti resume        -> Senti resumes all paused agents

/spec <description>  -> generates a structured spec breakdown with agent assignments
/review <pr-url>     -> tells the review agent to run Omar Gate on a specific PR
/test <flow>         -> tells the test agent to run AIdenID E2E on a specific flow
```

Plain natural language is also accepted: "Senti, what's happening?" or "@senti who modified
auth.js?" — Senti responds conversationally. The `/senti` prefix is for structured commands.

Example: Human types `/spec Build a notes API with auth and email verification` and the
system generates:

#### Why this is the killer feature

OpenClaw, CrewAI, AutoGen — they all orchestrate agents. But they're all **code-level
orchestration**. You write Python to define your agents and their interactions. SentinelLayer
sessions are **interface-level orchestration**. The human opens a web page, types natural
language, and coordinates agents that are already running in their own terminals with their
own tools. No code required. The agents joined via CLI. The human coordinates via dashboard.
Senti handles the plumbing.

This is the difference between writing a Slack bot and just typing in Slack.

## 12. Scale Architecture (1M+ Concurrent Sessions)

This system must handle millions of concurrent sessions. Schema and infrastructure are
designed for horizontal scale from day one.

### 12.1 Database: PostgreSQL with Partitioning

```sql
-- Sessions table partitioned by month
CREATE TABLE sessions (
    id VARCHAR(64) PRIMARY KEY,
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    status VARCHAR(20) NOT NULL DEFAULT 'active',
    created_at TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    expires_at TIMESTAMPTZ NOT NULL
) PARTITION BY RANGE (created_at);

CREATE TABLE sessions_2026_04 PARTITION OF sessions
    FOR VALUES FROM ('2026-04-01') TO ('2026-05-01');
CREATE TABLE sessions_2026_05 PARTITION OF sessions
    FOR VALUES FROM ('2026-05-01') TO ('2026-06-01');

-- Indexes:
--   (user_id, status)
--   (status, last_activity_at)
--   (created_at DESC)
--   (client_ip) for admin IP search

-- session_agents: hash-partitioned by session_id
-- session_errors: range-partitioned by created_at
-- session_findings_verdicts: hash-partitioned by session_id
```

### 12.2 Redis: Sharded by Session ID

```
session:{session_id}:events          (LIST, TTL = session TTL + 1h)
session:{session_id}:agent:{id}:heartbeat  (STRING, TTL = 120s)
session:{session_id}:human_messages  (LIST, TTL = session TTL)

Sharding: CRC16(session_id) % num_shards
1M sessions * ~100 events = ~100M keys
Redis Cluster with 8-16 shards handles this.
```

## 15. E2E Encryption

All session data is encrypted at rest. Stream events, blackboard entries, and agent snapshots
are encrypted before writing to disk.

### 15.1 Encryption scheme

```javascript
// Pattern: src/auth/session-store.js (AES-256-GCM with per-session key)

// On session create:
// 1. Generate session encryption key: crypto.randomBytes(32)
// 2. Store key in OS keyring: keytar.setPassword('sl-session', sessionId, key.toString('base64'))
// 3. If keyring unavailable: encrypt key with user's credentials.key and store in metadata

// On stream append:
// 1. Load session key from keyring (or decrypt from metadata)
// 2. Encrypt event JSON: AES-256-GCM with random 12-byte IV
// 3. Write: base64(iv) + "." + base64(ciphertext) + "." + base64(tag) + "\n"

// On stream read:
// 1. Load session key
// 2. Split line on "."
// 3. Decrypt: AES-256-GCM with IV and tag
// 4. Parse JSON

// On S3 archive:
// Session key is included in the archive (encrypted with a master archive key
// stored in AWS Secrets Manager). Training data pipeline decrypts with master key.
```

### 15.2 `src/session/crypto.js`

```javascript
// Pattern: src/auth/session-store.js (encryptToken / decryptToken)

export function encryptEvent(eventJson, sessionKey)
export function decryptEvent(encryptedLine, sessionKey)
export async function loadSessionKey(sessionId)
export async function createSessionKey(sessionId)
```

### 15.3 Why E2E encryption matters

- Session streams contain agent communications, findings, file paths, and codebase context.
- If `.sentinelayer/` is accidentally committed or shared, stream content is unreadable.
- S3 archives are double-encrypted: session key (AES-256-GCM) + AWS SSE.
- Only agents with access to the local keyring can read the stream.
- Remote agents (future) use key exchange via SentinelLayer API.

## 16. Security Constraints

- Session files live in `.sentinelayer/sessions/` which is in `.gitignore` — never committed.
- All stream data is AES-256-GCM encrypted at rest with per-session keys.
- Session keys stored in OS keyring (preferred) or encrypted file fallback.
- No tokens, secrets, or credentials in stream events — agent IDs are opaque short identifiers.
- Session TTL default: 24 hours, auto-renewable to max 72 hours.
- S3 archives double-encrypted: session key (AES-256-GCM) + AWS SSE-S3.
- Stream append is file-locked: no corruption from concurrent writers.
- Senti LLM calls use existing `sentinelayer_token` via proxy — no new credentials.
- `sl session say` sanitizes message: strip control characters, truncate to 2000 chars,
  reject if contains credential patterns (`sk-`, `ghp_`, `Bearer `).
- Agent IDs are not predictable — 4-char random hex suffix prevents enumeration.
- Sessions are scoped to a single codebase path — no cross-repo access.
- Notification files contain only unread count + summary, not full message content.
- Per-tool risk class + allowlist enforced (SWE §K.1).
- OAuth `resource` binding + audience validation on MCP tools (SWE §K.2).
- Token passthrough forbidden on MCP tools (SWE §K.2).
- SSRF protections on all URL/network tools (SWE §K.1).

## 19. Handoff Notes (for Codex)

These are the non-negotiable operating rules for implementing this spec.

1. **Start at PR 0.** Do not jump ahead. The event schema is the foundation every later PR
   depends on. Every emitter must use `createAgentEvent()` before any session store work.

2. **Use the Omar loop per CLAUDE.md.** Every PR is branched, verified, gate-checked, and
   merged only when green. The loop is:
   ```
   git checkout main && git pull --ff-only
   git checkout -b roadmap/pr-<n>-<slug>
   # implement
   npm run verify
   node bin/create-sentinelayer.js /omargate deep --path . --json
   node bin/create-sentinelayer.js /audit --path . --json
   git push -u origin <branch>
   gh pr create --fill
   gh run list --workflow "Omar Gate" --branch <branch> --limit 1 \
     --json databaseId --jq ".[0].databaseId"
   gh run watch <runId> --exit-status
   # merge only when P0=0, P1=0, P2<=5
   ```
   No shortcuts. No `--no-verify`. No amending to bypass hooks.

3. **Cite SWE framework gates in every PR description.** Every PR description includes a
   three-bullet evidence block listing which SWE framework §E.3 red flags the PR mitigates,
   with a file or line link for each. Reviewers will reject PRs that skip this block.

4. **Do not skip the eval regression step.** Any PR that touches a prompt, policy, model
   route, or tool allowlist runs the eval suite. Regression blocks the merge (SWE §I.2).
   If no eval suite exists for the affected scope, author one in the same PR — do not ship
   prompt changes blind.

5. **Preserve investor-grade evidence.** Every release artifact carries SBOM and cosign
   attestation. Every autonomous remediation run emits the full ODCP artifact chain
   (`observability/<date>/<work_item_id>/*`). Treat these as non-negotiable.

6. **Kill switches are mandatory, not optional.** Every PR that introduces or extends an
   autonomous write path adds a test that exercises `sl session kill --agent <id>` or
   `sl session kill --all`. No merge without this test.

7. **Keep ODCP folded in, not bolted on.** The nine ODCP services fit into the existing PR
   sequence (2.5, 3, 3.5, 4, 10, 13, 16). Do not create a parallel ODCP batch. Do not
   duplicate code between `src/daemon/` and `src/session/` — extend the existing daemon
   modules.

8. **Respect AsyncLocalStorage isolation + worktree isolation.** Every write-capable
   persona runs inside `src/utils/swarm/spawnInProcess.ts` ALS context AND inside a worktree
   (`src/tools/AgentTool/AgentTool.tsx:99-100, 582-593`). Do not bypass either layer.

9. **Senti is the only daemon identity.** Do not reintroduce older names (e.g., Kai, Nexus)
   anywhere — not in code, not in docs, not in test fixtures. Senti is short for
   SentinelLayer and is the canonical moderator.

10. **Report evidence, not claims.** Every PR links the specific artifact that proves each
    acceptance criterion: `npm run verify` output, Omar Gate run URL, test file path, event
    schema sample, artifact chain SHA. Reviewers verify artifacts before signing off.
